**Reason:** react-doctor flagged `rules-of-hooks` at `transitions.ts:25` — a `withReducedMotion()` plain function that called `useReducedMotion()` (a hook) inside it. This violation was NOT in the Batch 1 plan (which only identified `BudgetConfirmation.tsx` as the rules-of-hooks source).

**Impact:** Low. Dead code deleted. No functionality affected.

### AuthCallback.tsx — cleanup pattern

The `timeoutIds.forEach(clearTimeout)` cleanup was rejected by react-doctor's static analysis even though semantically correct. The fix used an explicit `for...of` loop with `clearTimeout(id)` calls, which the analyzer recognized. Behavior is identical.

## 5. ARCHITECTURE NOTES

### BudgetConfirmation.tsx — hooks-before-guard pattern
The corrected order is: all hooks → guard clause → derived values. The `recommendation?.budgetItems ?? []` fallback ensures the hook initializes with an empty array when `recommendation` is undefined (which only happens transiently before the Navigate redirect fires). This is safe because the rendered content after the guard always reads `editedItems` which is derived from the actual `recommendation.budgetItems` on the successful path.

### AuthCallback.tsx — async effect cleanup strategy
The `delay()` helper pattern collects all timeout IDs into a shared array, ensuring cleanup covers both the polling delays (`await delay(500/300)`) and the navigate timeouts. Using `for...of` with an explicit `clearTimeout(id)` call is more readable and satisfies static analyzers that don't track `forEach` callbacks.

### Laporan.tsx — component hoisting
`CustomTooltip` was a stable component (no closure over `LaporanContent` state or props) — moving it to module scope required zero changes to its body. React now sees a stable reference for the tooltip on every `LaporanContent` render, preventing unnecessary remounts.
