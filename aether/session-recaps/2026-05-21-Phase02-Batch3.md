## 3. ARCHITECTURE NOTES

### Principle 03 (Token-Driven Styling) Applied
- ButtonBase: All 4 variant colors use CSS variables (primary: `--color-brand-primary`, secondary: `--color-border-neutral`, ghost: `--color-content-secondary`, danger: `--color-sentiment-negative`)
- InputBase: All semantic colors use CSS variables (background: `--color-bg-card`, border: `--color-border-neutral`, error: `--color-sentiment-negative`)
- No hardcoded hex values in either component — all styling references theme.css tokens

### Principle 06 (Accessibility) Implemented
- ButtonBase: Focus ring (ring-[3px]) using brand primary color, keyboard navigation via ButtonHTMLAttributes, disabled state with opacity-40, loading state prevents interaction
- InputBase: Focus ring matches ButtonBase pattern, semantic label element, error text marked with color token, disabled cursor-not-allowed, proper input type handling
- Touch targets: Both components meet 44×44px minimum (ButtonBase default size, InputBase h-11)

### Design System Alignment
- Consistent spacing: Tailwind scale (px-3, py-2, gap-2) throughout
- Consistent radius: Both use `rounded-[var(--radius-md)]` from theme.css
- Consistent transitions: ButtonBase uses `duration-[var(--duration-fast)]` from transitions.ts
- Consistent motion: InputBase focus-within transition uses `duration-[var(--duration-fast)]`

### Currency Input Design Choice
- Why formatRupiah in component vs. utils: Currency formatting is input-specific behavior; kept logic in InputBase for reusability
- Raw numeric emission: onChange emits numeric string to allow form libraries (react-hook-form) to work seamlessly
- Focus/blur UX: Showing raw number on focus reduces friction for editing amounts; reformatting on blur provides visual feedback
- State management: Used local useState for display formatting (simpler than prop drilling for currency-mode-only state)

### Component Composition Strategy
- ButtonBase uses CVA for variant management (consistent with shadcn patterns in project)
- InputBase uses inline Tailwind with conditional className logic (simpler for single-component styling, no CVA needed)
- Both use forwardRef for proper ref forwarding to underlying elements
