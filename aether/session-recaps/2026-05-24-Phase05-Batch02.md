## 3. ARCHITECTURE NOTES

### chatStore.ts was NOT modified
`chatStore.ts` already sends `conversationHistory` correctly. The gap was entirely on the server side (`ask-assistant.js` ignoring it). No changes to the store were needed.

### ChatMessage.sender → Anthropic role mapping
`ChatMessage` uses `sender: 'user' | 'assistant'`, not `role`. The mapping is:
```javascript
m.sender === 'user' ? 'user' : 'assistant'
```
This maps any non-user sender (currently only `'assistant'`) to `'assistant'`. Safe.

### Final messages array structure
```javascript
messages: [
  // Prior turns (up to 10), mapped from ChatMessage[]
  { role: 'user', content: 'Ringkasan bulan ini' },
  { role: 'assistant', content: 'Berikut ringkasan...' },
  // ... more history ...
  // Current turn — financial context + question
  { role: 'user', content: contextStr },  // contextStr includes income/expense data + question
]
```
History messages contain raw user questions and raw AI responses (no financial context wrapping). The current turn always has full financial context via `contextStr`. This is intentional — historical turns are lightweight references; the current turn is fully contextualized.

### Backward compatibility
If `conversationHistory` is absent, `null`, or not an array, `historyMessages = []` and the call is functionally identical to the pre-Batch-2 behavior.

### API schema confirmed for Batch 3
- POST body: `{ question, transactions, month, year, conversationHistory? }`
- Response: `{ response: string }`
- The `data.response || data.answer` fallback in chatStore line 124 remains valid.

### "Bersihkan Chat" button behavior
`clearMessages()` from chatStore resets: `messages: []`, `suggestedActions: []`, `error: null`, `isLoading: false`. Since Zustand `persist` is active, the cleared state is also written to localStorage — the chat will be empty on next page load.
