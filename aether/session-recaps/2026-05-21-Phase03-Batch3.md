## DETAILED IMPLEMENTATION

### 1. Layout.tsx Refactor (~80 lines)

**Purpose:** Replace monolithic 501-line layout with clean component composition.

**Key Changes:**

1. **Component Composition** (instead of inline HTML)
   - Removed inline nav definitions (navItemsGrouped, navItems)
   - Removed hardcoded sidebar markup
   - Removed old MobileNav integration
   - Added: DesktopSidebar, HeaderBar, SectionTabBar, BottomNavigation, MobileFilterSheet

2. **Responsive Layout Structure**
   - Mobile (<768px): HeaderBar(mobile) + SectionTabBar + Outlet + BottomNavigation
   - Tablet (768-1023px): DesktopSidebar(rail 60px) + HeaderBar(desktop) + SectionTabBar + Outlet
   - Desktop (1024px+): DesktopSidebar(full 200px) + HeaderBar(desktop) + SectionTabBar + Outlet
   - Main content margin: `md:ml-[60px]` (tablet rail) + `lg:ml-[200px]` (desktop)

3. **Responsive Component Rendering**
   - Tablet sidebar: `hidden md:block lg:hidden` + `isRailMode={true}`
   - Desktop sidebar: `hidden lg:block` + `isRailMode={false}`
   - Mobile header: `lg:hidden` + `variant="mobile"` + filter sheet toggle
   - Desktop header: `hidden lg:block` + `variant="desktop"`
   - BottomNavigation: Mobile-only, rendered unconditionally (has its own lg:hidden class)
   - MobileFilterSheet: Rendered unconditionally, only usable on mobile

4. **State Management**
   - Local state: `isTransactionModalOpen`, `isFilterSheetOpen`
   - Removed: `isMobileMenuOpen`, `selectedMonth`, `isHidden` (now in respective contexts/hooks)
   - TransactionModal FAB button with spring animation (whileHover, whileTap)

5. **Accessibility & Usability**
   - Skip-to-main-content link preserved
   - `id="main-content"` anchor for semantics
   - FAB button `aria-label="Tambah transaksi"` (Indonesian)
   - Main content padding: `pb-20 md:pb-6` (to avoid overlap with bottom nav on mobile)

## ARCHITECTURE DECISIONS

### Decision 1: Component Composition vs. Inline HTML
**Choice:** Extract all nav logic into dedicated components  
**Rationale:** Layout.tsx responsibility narrows to composition; each component owns its own responsive behavior  
**Tradeoff:** Two HeaderBar renders and two DesktopSidebar renders (for different breakpoints), but CSS ensures only one renders at any time

### Decision 2: Dual DesktopSidebar Instances for Rail Mode
**Choice:** Render DesktopSidebar twice (one with isRailMode={true}, one with isRailMode={false}), hidden conditionally  
**Rationale:** Avoids complex state/media-query logic in Layout; each instance is simple and predictable  
**Tradeoff:** Minimal performance impact (CSS hides one instance); future optimization could make DesktopSidebar self-aware of breakpoints

### Decision 3: All Redirects in Single Route Structure
**Choice:** Define 19 old-path redirects alongside new routes  
**Rationale:** Explicit, discoverable, traceable; easier to verify completeness  
**Tradeoff:** Adds routing surface area; alternative would be middleware redirect layer

### Decision 4: NavigationProvider as Innermost Provider
**Choice:** Wrap Routes in NavigationProvider (inside MonthFilterProvider)  
**Rationale:** Navigation context doesn't depend on other providers; natural hierarchy  
**Tradeoff:** Could be moved to outer layer if future features require it

### Decision 5: Defer Link Updates to Batch 4
**Choice:** Don't bulk-search and update internal links; treat as separate concern  
**Rationale:** 5-file limit reached (Layout.tsx, App.tsx); searching would add many more files  
**Tradeoff:** If internal links exist (low probability based on grep), users see redirects (functional but not ideal UX)
