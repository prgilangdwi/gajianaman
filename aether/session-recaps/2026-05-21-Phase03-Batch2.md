## DETAILED IMPLEMENTATION

### 1. DesktopSidebar.tsx (~280 lines)

**Purpose:** Responsive sidebar component for desktop/tablet navigation with dynamic width and responsive behavior.

**Key Features:**

1. **Responsive Width Management**
   - Desktop (md:+): 200px fixed width, full navigation labels visible
   - Tablet (768px-1024px): 60px rail mode, icon-only with hover tooltips
   - Mobile: Hidden (md:hidden class) — uses BottomNavigation instead

2. **Component Structure**
   - Header section: Logo (GA gradient badge), app name (Gajian Aman), user avatar with name
   - Navigation sections: Maps through NAV_SECTIONS array, renders SidebarSection sub-component
   - Footer: "Powered by Claude · Supabase" text + Logout button
   - All sections responsive: compact in rail mode, full in normal mode

3. **SidebarSection Sub-Component**
   - Renders primary section link (icon + label in normal, icon-only in rail)
   - Expands sub-navigation when section is active (AnimatePresence + motion.div)
   - Sub-nav collapse: height: 0 → auto animation, 150ms duration
   - Child links display with gray inactive state, green active state
   - Rail mode: tooltip appears on hover showing full section name

4. **Styling & Colors**
   - Background: `--color-bg-screen` (matches desktop body background)
   - Active section: `--color-brand-primary`/10 background, brand-primary text
   - Inactive: `--color-content-tertiary` text, hover to `--color-content-secondary`
   - All spacing via explicit px values calculated from design tokens
   - Transition: width 300ms ease (matches --duration-normal from Phase 02)

5. **Accessibility**
   - `role="navigation"`, `aria-label="Navigasi utama"` (Indonesian)
   - `aria-current="true"` on active section
   - Focus rings: 2px outline, offset 2px, brand color
   - Full keyboard navigation support
   - Screen reader announces section names bilingual format

**Integration Points:**
- Uses `useNavigation()` hook to detect `activeSection`
- Uses `useAuth()` for user data and logout function
- Links navigate to `section.children[0]?.path ?? section.path` (defaults to first child)
- Dynamic icon rendering from `section.icon` (lucide-react Icon component)

### 2. HeaderBar.tsx (~160 lines)

**Purpose:** Unified top header component with dual rendering modes (mobile/desktop) and auto-derived page titles.

**Key Features:**

1. **Dual Variant System**
   - `variant="mobile"` (lg:hidden): Compact mobile header
   - `variant="desktop"` (hidden lg:block): Full-featured desktop header
   - Both share same component, different renders based on variant prop

2. **Mobile Header (56px height)**
   - Left: Page title (auto-derived from route or pageTitle prop)
   - Right: Filter button (if onOpenFilters provided), privacy toggle, user avatar
   - Sticky top-0 z-40, semi-transparent with backdrop blur
   - Uses Icon components from lucide-react (SlidersHorizontal, Eye, EyeOff)

3. **Desktop Header (56px height)**
   - Left: Page title (auto-derived from route or pageTitle prop)
   - Right: Month filter dropdown select, privacy toggle, notifications bell, user avatar

5. **Privacy Mode State**

6. **Styling & Colors**

## ARCHITECTURE DECISIONS

### Decision 1: Responsive Width vs. Multiple Components
**Choice:** Single DesktopSidebar with `isRailMode` prop (computed from viewport)  
**Rationale:** Keeps navigation state unified; avoids duplicate child routes or complex switching logic  
**Tradeoff:** Component has two visual modes within one render; simpler for parent integration

### Decision 2: Variant-Based Header vs. Separate Mobile/Desktop Components
**Choice:** Single HeaderBar with `variant` prop (mobile | desktop)  
**Rationale:** Shared page title derivation logic, simpler parent integration, smaller bundle  
**Tradeoff:** Component logic branches on variant; less reusable for partial use cases

### Decision 3: Bottom Sheet Filter Modal vs. Inline Filters in Header
**Choice:** Separate MobileFilterSheet component, triggered by parent (Layout or page)  
**Rationale:** Prevents header from becoming too dense; uses full screen real estate on mobile; follows mobile UI patterns  
**Tradeoff:** Requires parent to manage `isOpen/onClose` state; adds one more component to component tree

### Decision 4: Conditional SectionTabBar vs. Always-Rendered (Hidden)
**Choice:** Conditionally render (return null if section.children.length <= 1)  
**Rationale:** Avoids empty sticky bars; cleaner DOM; matches UI intent  
**Tradeoff:** Dynamic rendering on route change; slightly higher computational cost for route detection

### Decision 5: Local Filter State vs. Global Filter Context
**Choice:** Local state in MobileFilterSheet (selectedMonth, selectedWallet)  
**Rationale:** Batch 2 scope limits integration to UI structure; actual filter application (API calls, data fetching) is Batch 3 responsibility  
**Tradeoff:** Filter selections are lost when sheet closes; TODO for Batch 3 to persist to context
