## 3. ARCHITECTURE NOTES

### API Schema Deviation (CRITICAL — Batch 2 Architect must know)
The `runtime-phase-05.md` Section 4 code pattern (and staging doc) shows the OLD `ask-assistant.js` API:
```
POST body: { message: string, userId: string }
Response:  { reply: string }
```
**This is INCORRECT for the current implementation.** The Executor prompt correctly documented the actual schema:
```
POST body: { question: string, transactions: Transaction[], month: number, year: number }
Response:  { reply: string }   ← wait, actually { response: string }
```
**The actual response field is `data.response` (not `data.reply`).** This was verified by the Executor prompt's API SCHEMA section which noted the staging doc was wrong. The `chatStore.ts` uses `data.response` — this implementation mirrors that exactly.

**Batch 2 Architect must verify `ask-assistant.js` actual response field before generating the Batch 2 prompt.** If Batch 2 adds `messages[]` array support, backward compatibility must use `data.response` not `data.reply`.

### ChartInsight placement
`ChartInsight` is placed directly inside `Forecasting()` return, between `</Tabs>` and `</motion.div>`. This is intentional — it must NOT be inside `renderMonthlyView()` or `renderWeeklyView()` because those are standalone functions outside the component and cannot access hook state (`forecastInsight`, `insightLoading`). The insight therefore renders below both tabs regardless of which tab is active, which is the correct UX.

### `useTransactions` double auth call
`Forecasting.tsx` already calls `useAuth()` at line 56. `useTransactions` also calls `useAuth()` internally. React context ensures both get the same value — no duplicate network call.

### Silent fallback behavior
On fetch error or timeout (5s `AbortSignal.timeout`), `forecastInsight` remains `''`. ChartInsight renders `error={!insightLoading && !forecastInsight}` which shows a static fallback string. The forecast charts are completely unaffected.
