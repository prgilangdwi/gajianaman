### 1. `DashboardSkeleton.tsx` (Skeleton Loader)

**Purpose:** Layout-preserving skeleton that matches exact dashboard section heights and structure, preventing layout shift during data loading.

**Design Approach:**
- Matches the exact visual hierarchy of the final dashboard layout
- Each skeleton placeholder sized to match its final content height
- Uses `animate-pulse` class for subtle loading animation
- Fully accessible with `aria-busy="true"` and `aria-label="Memuat dashboard..."` (Loading dashboard...)

|---------|-----------|--------|---------|

**Styling:**

- `border` + `rounded-[var(--radius-*)]` for consistent corner radius
- No hardcoded colors or dimensions
- Nested grid layouts (3 columns for multi-card sections)

**Accessibility:**
- `aria-busy="true"` tells screen readers content is loading
- `aria-label="Memuat dashboard..."` provides localized loading context

### 2. `CollapsibleSection.tsx` (Generic Collapsible Wrapper)

**Purpose:** Reusable progressive disclosure component for collapsible sections (Goals, Categories, Recent Transactions, etc.). Enables users to expand/collapse content while preserving state and providing smooth animations.

**Features:**

**State Management:**
- `useState(defaultOpen)` tracks open/closed state
- `defaultOpen` prop allows controlling initial state (default: true)
- Simple toggle logic via `setIsOpen((prev) => !prev)`

**UI Elements:**
- Title with optional icon (left-aligned)
- Count badge in gray text: `({count})` format (e.g., "Goals (3)")
- Chevron indicator: `ChevronUp` when open, `ChevronDown` when closed
- Optional className for external styling

**Keyboard Accessibility:**
- `onKeyDown` handler captures Enter and Space keys
- Prevents default scrolling behavior
- Triggers same toggle logic as click
- `aria-expanded={isOpen}` announces state to screen readers

**Animation System:**
```typescript
initial={ prefersReduced ? { opacity: 0 } : { height: 0, opacity: 0 } }
animate={ prefersReduced ? { opacity: 1 } : { height: 'auto', opacity: 1 } }
exit={ prefersReduced ? { opacity: 0 } : { height: 0, opacity: 0 } }
transition={{ duration: 0.15 }}
```
- Height-based animation for smooth content expansion (when motion not reduced)
- Opacity-only animation when user prefers reduced motion
- `AnimatePresence` handles enter/exit lifecycle
- 150ms transition duration for snappy feel

**Styling:**
- Button: `flex items-center gap-2 w-full text-left py-2 group`
- Title: `text-sm font-semibold` with `textColorVar('content-primary')`
- Count: `text-xs font-normal` with `textColorVar('content-tertiary')`
- Chevron: `textColorVar('content-tertiary')` with hover to `content-secondary`
- Hover effect: `group-hover:text-[var(--color-content-secondary)]` on chevron

**Motion Preference:**
- Respects `useReducedMotion()` hook
- Reduces motion when user has `prefers-reduced-motion: reduce` CSS media query set
- Ensures accessibility for users sensitive to animations

**Lines of Code:** ~93

## ARCHITECTURE DECISIONS

### Decision 1: DashboardSkeleton Section Heights
**Choice:** Match exact component heights (h-28, h-20, h-16, h-64, h-14)  
**Rationale:** Layout shift prevention (CLS metric). Users see exact shape before content loads. No surprise jumps.  
**Tradeoff:** If final content heights change in Batch 3, skeleton must be updated. Acceptable since this is a known sync point.

### Decision 2: Collapsible via State Hook
**Choice:** Use `useState(defaultOpen)` instead of compound component pattern  
**Rationale:** Simplicity. Fewer prop-drilling issues. Self-contained toggle logic. Works with any children.  
**Tradeoff:** No external state control (parent cannot programmatically toggle). Acceptable for dashboard use case where each section manages its own state.

### Decision 3: Motion Preference Integration
**Choice:** Use `useReducedMotion()` hook to branch animation styles  
**Rationale:** WCAG AAA compliance. Users with vestibular disorders benefit from opacity-only transitions. Respects system preference.  
**Tradeoff:** Requires custom hook (already available). Two separate animation configs (slight code duplication in initial/animate/exit). Acceptable for accessibility priority.

### Decision 4: Defer Overview.tsx to Batch 3
**Choice:** Stop after support components, do NOT compose into Overview yet  
**Rationale:** Batch 2 scope is support components only. Composing Overview requires all 6 components + layout orchestration (8+ steps). Separating concerns keeps sessions focused.  
**Tradeoff:** Dashboard components exist but unused. Intentional per governance rules. Batch 3 will integrate everything.
