## 3. ARCHITECTURAL DECISIONS

### Code Splitting Strategy
- **Approach:** Route-based lazy loading with Suspense fallback
- **Why:** Each page is heavy (10-35KB gzipped); lazy loading defers parsing until route access
- **Trade-off:** Small UX delay on first page load (LoadingState shown) vs. massive initial bundle reduction

### Recharts Tree-Shaking
- **Approach:** Removed unused chart component imports from pages
- **Why:** Recharts is large (~150KB gzipped); tree-shaking removes dead code during Vite build
- **Validation:** Verified each remaining import is actually used in the JSX

### Image Lazy Loading Strategy
- **Approach:** Default `loading="lazy"` on all images in ImageWithFallback
- **Why:** Images are often below the fold (receipts, avatars); deferring load improves LCP
- **Override support:** Callers can pass `loading="eager"` if above-the-fold

### Pre-Existing Import Path Fixes
- **Issue:** Phase 08 created new components in nested folder structure (`components/features/transactions/`) but used relative imports assuming old flat structure
- **Root cause:** Old codebase is at `src/app/components/`, but ExpandableTransactionRow is at `src/components/features/...` — paths didn't match
- **Fix:** Switched all imports to use `@` alias which resolves to `./src` (tsconfig.app.json baseUrl)
- **Impact:** Fixes were required to make React.lazy() work (blocking build errors under the new structure)
