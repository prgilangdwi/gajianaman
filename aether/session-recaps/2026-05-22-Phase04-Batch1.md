## DETAILED IMPLEMENTATION

### 1. `useDashboardData.ts` (Orchestration Hook)

**Purpose:** Single hook combining 12+ data sources into one unified `DashboardData` interface. Eliminates the need for Overview.tsx to call multiple hooks directly, reducing component complexity and improving testability.

**Data Sources Combined:**
- `useAuth()` — user profile
- `useMonthFilter()` — current month/year context
- `useWalletFilter()` — active wallet filter
- `useTransactions()` — primary transactions + derived income/expenses
- `useRecentTransactions()` — last 10 transactions
- `useBudgets()` — monthly budget data
- `useGoals()` — savings goals
- `useWallets()` — wallet list
- `useRecurringBills()` — upcoming bills
- `useInsights()` — transaction insights
- `useFinancialHealth()` — health metrics
- `useFinancialHealthScore()` — health score + label
- `useBudgetRecommendations()` — budget recommendations
- `useAIInsights()` — AI-generated insights + loading state

**Key Derived Calculations (all memoized):**
- `balance` = income - expenses
- `availableToSpend` = Math.max(0, totalBudget - expenses)
- `budgetRemainingPct` = percentage remaining
- `daysRemaining`, `dailyBudget`, `daysUntilPayday` — time-based calculations
- `topCategories` — top 3 expense categories sorted by amount
- `activeGoals` — goals < 100% progress, sorted descending
- `weeklyChartData` — 4-week income vs expense breakdown
- `greeting` — time-of-day greeting ("Selamat pagi/siang/sore")

**Type Export:**
```typescript
interface DashboardData {
  user, greeting, income, expenses, balance, availableToSpend, ...
}
```

**Performance Considerations:**
- All derived values wrapped in `useMemo()` with proper dependency arrays
- Wallet filtering applied at hook level (filtered, filteredIncome, filteredExpenses)
- Chart data aggregated weekly (4 weeks, not daily) to reduce memory footprint
- Top categories limited to top 3 (sorted by amount, percentage calculated)

**Lines of Code:** ~360

### 2. `StatCard.tsx` (Metric Card Component)

**Purpose:** Reusable metric display card with icon, label, formatted amount, and optional trend indicator. Used throughout dashboard (hero balance, status row, etc.).

**Features:**
- **Variants:** `default` (standard), `hero` (oversized), `compact` (minimal)
- **Trend Indicator:** Optional `{ direction: 'up'|'down'|'flat', value: string }`
- **Privacy-Aware:** Uses `PrivacyAmount` wrapper for sensitive numbers
- **Styling:** All via CSS variables (`--color-*`, `--radius-*`, `--shadow-*`)
- **Accessibility:** `role="group"`, `aria-label={label}`

**Responsive:**
- Mobile: `p-4`, `text-lg` value
- Hero: `p-5`, `text-2xl sm:text-3xl` value
- Compact: `p-3`, `text-sm` value

**Dependencies:**
- `lucide-react` — TrendingUp, TrendingDown, Minus icons
- `PrivacyAmount` — blur amount based on privacy setting
- `@/lib/utils` — cn(), textColorVar(), bgColorVar(), borderColorVar()

**Lines of Code:** ~80

### 3. `ChartCard.tsx` (Chart Wrapper Component)

**Purpose:** Reusable wrapper for Recharts with consistent styling, collapsible sections, and proper responsive behavior. Eliminates duplication of chart layout patterns.

**Features:**
- **Collapsible:** Optional `collapsible` prop with defaultCollapsed state
- **Responsive:** Uses `ResponsiveContainer` from Recharts
- **Animation:** Smooth collapse/expand using `motion/react` with `useReducedMotion()` support
- **Empty State:** Shows contextual message when `isEmpty={true}`
- **Header:** Title + subtitle + optional action button

**Default Height:** 220px (customizable via `height` prop)

**Styling:**
- Card uses `--color-bg-card`, `--color-border-neutral`
- Collapsible chevron animates with 200ms duration
- Respects `prefers-reduced-motion` CSS media query

**Dependencies:**
- `recharts` — ResponsiveContainer
- `motion/react` — AnimatePresence, motion.div
- `lucide-react` — ChevronDown, ChevronUp

**Lines of Code:** ~120

### 4. `InsightCard.tsx` (AI Insight Component)

**Purpose:** Display AI-generated insights with severity-based styling (info, warning, critical, success). Triple-coded (icon + color + text) following WCAG AAA accessibility principles.

**Severity Levels:**
| Level | Icon | Background | Border | Use Case |
|-------|------|-----------|--------|----------|
| `info` | Sparkles | `--color-brand-primary-light` | `--color-brand-primary` | Tips, features |
| `warning` | AlertTriangle | `--color-sentiment-warning-bg` | `--color-sentiment-warning` | Near limits, caution |
| `critical` | AlertTriangle | `--color-sentiment-negative-bg` | `--color-sentiment-negative` | Over budget, urgent |
| `success` | CheckCircle | `--color-sentiment-positive-bg` | `--color-sentiment-positive` | Goals met, positive |

**Features:**
- Optional emoji override (replaces icon)
- Optional title + body text
- Left border accent (3px)
- Flexible width with min-width-0 for text overflow

**Accessibility:**
- `role="status"` announces insights to screen readers
- `aria-label` provides context
- Icon + color + text (never color-only)

**Dependencies:**
- `lucide-react` — Sparkles, AlertTriangle, Info, CheckCircle

**Lines of Code:** ~75

## ARCHITECTURE DECISIONS

### Decision 1: Orchestration Hook Pattern
**Choice:** Create `useDashboardData()` hook combining 12+ data sources  
**Rationale:** Prevents Overview.tsx from having a massive hook imports list (would be 15+ lines). Consolidates all derived calculations in one place. Makes testing easier (mock one hook instead of many).  
**Tradeoff:** Hook is ~360 lines, but that's acceptable for a single responsibility (data orchestration). Could be split further in future, but currently appropriate.

### Decision 2: Separate Dashboard Components Directory
**Choice:** Create `frontend/src/app/components/dashboard/` subfolder  
**Rationale:** Dashboard-specific components won't clutter global components directory. Barrel export provides clean API. Future phases (Spending, Analytics) may reuse these components.  
**Tradeoff:** One more directory level, but benefits clarity and organization.

### Decision 3: Variant-Based StatCard
**Choice:** Use `variant` prop ('default' | 'hero' | 'compact') instead of separate components  
**Rationale:** All three variants share the same logic (icon, label, value, trend). Props make it easy to compose in different contexts (hero metric, status row, category cards).  
**Tradeoff:** StatCard is slightly more complex, but more flexible than three separate components.

### Decision 4: Generic ChartCard Children
**Choice:** Accept `ReactNode` children instead of locked to Recharts components  
**Rationale:** Future proofs against chart library changes. Allows any content (not just charts). Compatible with all Recharts chart types (Area, Bar, Line, etc.).  
**Tradeoff:** Type is `children as any` in ResponsiveContainer, but acceptable trade-off for flexibility.

### Decision 5: Defer Overview.tsx Rewrite to Batch 3
**Choice:** Stop after creating components, do NOT modify Overview.tsx yet  
**Rationale:** Batch 1 focuses on foundational pieces. Overview rewrite requires composing all components (8+ steps). Separating concerns reduces session complexity and allows Batch 2 to add support components (Skeleton, Collapsible) before Overview uses them.  
**Tradeoff:** Overview is not usable yet, but that's intentional for Batch 1 scope.
