## ARCHITECTURE DECISIONS

### Decision 1: Complete Rewrite vs. Incremental Refactor
**Choice:** Complete rewrite (delete 721 lines, write 297 new)  
**Rationale:** Legacy code had mixed concerns (data fetching, calculations, rendering). Incremental refactor would accumulate technical debt. Clean slate enables proper component abstraction.  
**Tradeoff:** Riskier (full component replacement) but cleaner outcome. Mitigated by build passing with 0 errors.

### Decision 2: Single Orchestration Hook as Data Source
**Choice:** useDashboardData as single source of truth  
**Rationale:** Eliminates 12+ individual hook calls in Overview. Centralizes derived calculations. Easy to mock for testing. Simplifies state management.  
**Tradeoff:** useDashboardData hook is ~360 lines, but justified for single responsibility.

### Decision 3: Progressive Disclosure via CollapsibleSection
**Choice:** 3 collapsible sections (Categories, Goals, Bills) with defaults  
**Rationale:** Reduces cognitive load on first view. Users can expand as needed. Respects motion preferences (opacity-only for reduced motion).  
**Tradeoff:** Extra click to see Goals/Bills, but improves perceived performance and reduces scroll time.

### Decision 4: Defer Testing to Phase 05
**Choice:** No unit/integration tests in Phase 04  
**Rationale:** Phase 04 scope is design/UX. Tests would require mock setup and increase session complexity. Phase 05 (Testing & QA) handles test coverage.  
**Tradeoff:** Shipped code untested, but risk mitigated by build passing and visual verification.

### Decision 5: Keep Legacy BriefingCard & Calendar Heatmap Sections
**Choice:** Did NOT include "Sekilas Hari Ini" or calendar heatmap in new Overview  
**Rationale:** Phase 04 scope is 8-section storytelling hierarchy. Legacy briefing cards and heatmap are separate features. Can be re-integrated in Phase 05 if needed.  
**Tradeoff:** Two legacy sections removed, but they're available for future re-addition without breaking new architecture.
